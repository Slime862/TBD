# 挂机地牢 v0.1 文档索引

## 文档目的
这组文档用于整理当前项目的 v0.1 方向，先把产品目标、系统边界、读表结构和开发任务拆清楚，方便后续持续维护。

当前方向基于以下核心想法：
- 以“小窗挂机、上班也能挂着玩”为产品优势。
- 参考《暗黑地牢》的站位、阵容、饰品和高压战斗气质。
- 重点放在局外养成与编队配置，局内完全无玩家操作。
- 普通关强调长线抗压挂机，精英/Boss 关强调短回合高烈度承伤与爆发检查。

## v0.1 目标
v0.1 只做两个核心界面与一条最小可玩链路：
- 营地界面：选英雄、配技能、切职业分支、升级、换装备、看下一地点。
- 旅行界面：左侧英雄自动前进并进入战斗，完整跑通普通战、精英战、Boss 战的自动表现。

## 设计原则
- v0.1 优先做“可跑通、可扩展、可读表”，不追求一次把所有 meta 系统补齐。
- 站位、相邻影响、推拉位移、技能触发条件是核心，优先保证这几项架构不要做死。
- 局内不做手操，但要保留足够的观赏反馈，让挂机也有爽感。
- 内容量先小，系统骨架先清晰，避免早期为兼容未来堆太多分支。

## 当前阶段建议
- 先按 4 站位架构搭系统，但 v0.1 只开放 2 名可用英雄，剩余位置可先留空或锁定。
- 英雄、怪物、地点、技能、饰品都走读表，避免一开始把平衡数据写死在代码里。
- 战斗判定和表现解耦：先保证数值流程可验证，再逐步补慢镜、高亮、受击和技能名展示。

## 待确认
- v0.1 上阵人数是“2 人队伍”还是“4 个站位里仅开放 2 人”。
- 每名英雄的“2 个普攻、5 个技能、2 个职业分支”是各自独立池，还是两个英雄共享内容模板。
- v0.1 是否需要至少 1 个可实际战斗的 Boss 内容，还是先只把地点数据结构与预览界面做出来。
