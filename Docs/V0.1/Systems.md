# 系统拆分与架构建议

## 1. 设计目标
v0.1 的系统设计重点不是“功能做满”，而是做到：
- 数据驱动
- 逻辑清晰
- 易于验证
- 能继续扩到更多英雄、技能、怪物与地点

## 2. 推荐的系统分层

### 2.1 RefData 配置层
职责：
- 读取英雄、技能、怪物、地点、饰品等静态表格。
- 提供运行时查询入口。

特点：
- 不处理战斗状态。
- 不直接依赖具体 UI。

### 2.2 RuntimeData 运行时数据层
职责：
- 保存当前营地选择、英雄等级、已装备技能、分支、饰品、站位等状态。
- 负责“玩家当前持有什么”和“本次出征带了什么”。

特点：
- 可作为后续存档系统基础。
- 与静态表数据分离。

### 2.3 BattleSim 战斗判定层
职责：
- 处理行动顺序、技能选择、触发条件、目标选择、伤害治疗、位移、死亡、胜负。

特点：
- 不依赖具体动画。
- 尽量输出结构化事件给表现层。

### 2.4 BattlePresentation 表现层
职责：
- 接收战斗事件并播放移动、受击、高亮、慢镜和技能名。
- 维护旅行界面的可视反馈。

特点：
- 不反向改写核心判定。
- 表现失败不应影响战斗结果。

### 2.5 UI 层
职责：
- 营地界面展示与配置。
- 旅行界面展示波次与战斗过程。
- 读取运行时数据并驱动用户流程。

## 3. v0.1 必须拆出的系统

### 3.1 游戏流程系统
负责：
- 营地 -> 旅行 -> 战斗 -> 结算/返回营地 的状态切换。
- 当前地点、当前波次、当前战斗状态管理。

建议：
- 先做轻量状态机，不要把所有流程塞进一个大 MonoBehaviour。

### 3.2 营地系统
负责：
- 英雄选择
- 站位编辑
- 分支切换
- 技能配置
- 饰品配置
- 升级入口
- 地点预览与出发

边界：
- 这里只处理出战前决策，不负责战斗判定。

### 3.3 英雄养成与编队系统
负责：
- 英雄基础属性
- 等级成长
- 被动
- 当前普攻
- 当前技能组
- 当前职业分支
- 当前饰品
- 队伍站位

边界：
- “静态模板”来自读表。
- “当前配置”来自运行时数据。

### 3.4 技能与触发系统
负责：
- 技能定义
- 施法站位限制
- 目标站位限制
- 触发条件判定
- 技能优先级
- 技能效果列表执行

建议：
- 先做通用字段 + 效果表，不要为每个技能单独写脚本。
- 个别极特殊技能再考虑单独逻辑钩子。

### 3.5 站位与位移系统
负责：
- 4 个位置的占位关系
- 相邻判断
- 前后排判断
- 推拉位移
- 位移后重新排列

建议：
- 这是核心系统之一，尽量单独封装，不要散落在技能代码里。

### 3.6 怪物与波次系统
负责：
- 普通怪、精英怪、Boss 的静态定义
- 地点波次
- 波次内敌人组成
- 预览信息展示

建议：
- 地点和波次分表，方便后续复用怪物内容。

### 3.7 战斗表现系统
负责：
- 行动开始/命中/受击/死亡等表现事件
- 慢镜
- 全屏压暗
- 单位高亮
- 技能名飘字
- 旅行移动与进战切换

建议：
- 事件驱动，不要让表现层直接去算伤害。

## 4. v0.1 可暂时简化的系统
- 建筑升级系统可以先保留为“营地页签/区域”，不必一次做完整建筑树。
- 装备/饰品强化可以先只做装备与替换，不做强化链。
- 结算系统可以先只返回营地，不必做复杂收益页。
- AI 行为树没有必要，先用规则驱动技能触发即可。

## 5. 推荐的关键运行时对象
- `HeroRuntimeData`：玩家持有的英雄状态。
- `TeamLoadoutData`：本次出征编队与配置。
- `StageRuntimeData`：当前地点、当前波次、Boss 信息。
- `BattleUnitRuntime`：战斗内单个单位状态。
- `BattleContext`：当前战斗全局上下文。
- `BattleEvent`：判定层发给表现层的事件。

这些对象的作用是把“表数据”和“本局实际状态”分开。

## 6. 推荐的战斗事件类型
v0.1 建议至少准备这些事件：
- 行动开始
- 技能选定
- 技能释放
- 造成伤害
- 造成治疗
- 命中/闪避
- 位移
- 单位死亡
- 波次结束
- 战斗结束

有了这些事件，后续加动画、日志、战报都会轻松很多。

## 7. 与当前项目原型的衔接建议
当前项目已经有：
- 小窗挂机方向
- 战斗预览的基础 UI
- Windows 小窗透明与置顶逻辑

因此 v0.1 文档建议沿着这个方向扩展：
- 保留“小窗可持续观看”的产品特征。
- 用新的营地/旅行流程替换现有纯演示内容。
- 不急着推翻现有窗口方案，先把真正的战斗与配置接进去。

## 8. 架构上的风险提醒
- 如果技能效果直接写死在 UI 逻辑里，后续会很难扩表。
- 如果战斗表现和伤害结算不分层，后续补动画时很容易引发逻辑回归。
- 如果站位与位移逻辑分散在各技能脚本里，后期加怪物时会很痛苦。
- 如果地点波次不读表，后续平衡与内容扩充成本会很高。

## 9. 当前建议优先级
v0.1 实现顺序建议如下：

1. 先立读表结构与 RefData 类型。
2. 再把运行时编队数据与营地配置流打通。
3. 再做战斗判定最小闭环。
4. 最后补旅行表现和技能演出。

这个顺序能保证前期每一步都有可验证结果，不会先陷进表现细节。
